# Gemini Project Log: Elevator Automation

## Текущее состояние систем (SQL & SCADA)

### 1. Инфраструктура БД
*   **Стандарт подключения:** Использование `ODBC Driver 17` с обязательным флагом `TrustServerCertificate=yes`.
*   **Безопасность:** Создан пользователь `HMI_User` (pwd: 12345) с правами `db_owner` во всех базах.
*   **Синхронизация:** Все имена механизмов и их ID (`SlotId`) импортируются из эталонного `graph.json`.

### 2. Логирование форсировок (ForceLog)
*   **Логика:** Используется XOR-сравнение битовых масок (DWORD) для определения конкретного изменившегося бита.
*   **Категории:** 
    *   `STD`: Стандартные механизмы (10 битов).
    *   `DRY`: Зерносушилка (Dryer) (27 битов для 7 вентиляторов и горелок).
*   **Оптимизация:** Реализован кольцевой буфер на **8000 записей**.

### 3. Аудит операторов (OperatorLog)
*   **Функционал:** Фиксация действий через `LogOperatorAction(slotId, actionText)`.
*   **Имя пользователя:** В WinCC Unified используется `HMIRuntime.Resources.UserName`.
*   **Оптимизация:** Кольцевой буфер на **10 000 записей**.

### 4. Учет наработки (RunTime)
*   **Метод:** Процедура `UpdateMechStatus` обрабатывает пуски/остановки без использования триггеров.
*   **Данные:** Итоговые часы доступны через View `vw_RunTimeHours` (автоматический пересчет секунд в часы).

## Важные нюансы реализации
1.  **SSL Errors:** Все строки подключения в Python и JS должны содержать `TrustServerCertificate=yes;`, иначе локальный SQL Express обрывает связь.
2.  **Dryer Forces:** При обработке форсировок сушилки используется `BIGINT` в SQL, чтобы избежать переполнения при проверке 31-го бита (DWORD).
3.  **Table Update:** Для отображения логов в SCADA данные упаковываются в JSON-строки через `Force_DataString` и `Moto_TableDataString`.
4.  **Emergency Stop:** Внедрен алгоритм резкого останова (Fast Stop) всех механизмов маршрута при любой аварии (Fault).
